01 · Mission

An autonomous 5-drone swarm —
simulated today, flying tomorrow.

What We Are Building

A containerised, full-stack swarm system for coordinated perception, real-time mapping, and adaptive mission execution. Architecture splits cleanly between intelligence (Mothership) and edge sensing (Support Fleet).

Why This Way

Radio bandwidth is the hard constraint in any real outdoor swarm. The system is designed from day one to segment traffic by role, so heavy payloads (LiDAR, 4K optics) never saturate the shared RF channel.

ORYNTH / AEROLABAGENDA
02 · Swarm Doctrine

Mothership & Support Fleet

DRONE 0  ·  Mothership

  • Central routing hub & RF gateway
  • Global path planner
  • LiDAR aggregator / map builder
  • Command relay to ground station

DRONES 1–4  ·  Support Fleet

  • Decentralised edge workers
  • Configurable onboard detection per node
  • Local localisation & sensor redundancy
  • Formation-keeping around the Mothership

WHY ASYMMETRIC → The Mothership owns intelligence and aggregation. The Support Fleet owns sensing and redundancy. Segmenting roles keeps uplink bandwidth to the ground station inside a practical WiFi envelope.

What each layer does

bridgesGazebo ↔ ROS 2 (ground truth, cmd_vel, arm, IMU)
platformTranslates actuator cmds to the sim; ArduPilot is a separate future backend
estimatorPublishes pose + TF chain earth → map → odom → base_link
controllerPID: motion reference → velocity setpoints
behaviorsAction servers for high-level primitives

Every drone runs an identical copy of this stack. Co-ordination is a separate orchestrator node that sends goals to all five via their namespaced action servers.

ORYNTH / AEROLABPHASE 01 — DONE
04 · Phase 1 Deliverables

Five things that work today Complete

01 · Containerised environment

Dockerfile extends aerostack2/nightly-humble. Adds PyTorch, Ultralytics, OpenCV-headless and both RMWs (CycloneDDS + Zenoh). GPU + X11 passthrough for Gazebo rendering. Build time: ~3–5 min.

02 · One-command swarm bringup

swarm_bringup_launch.py starts the Gazebo world and all five drone stacks in a single command. Each stack: bridges, platform, estimator, controller, five behaviour action servers.

03 · Teleoperation mode

Keyboard GUI controls drone0. Drones 1–4 follow in real-time diamond formation via FollowReferenceBehavior tracking drone0/base_link.

04 · Automated formation mode

flock_orchestrator arms, offboards, and takes off all five drones in parallel threads, then attaches the FollowReference module on drones 1–4. No polling loop.

05 · Validation test + dual-RMW support

mission_test.py flies drone0 through a waypoint and asserts followers hold their offsets within 0.5 m. rmw:=zenoh argument swaps middleware at launch — includes router startup, session config, and a tf_static replay workaround.

ORYNTH / AEROLAB04 — REPO
05 · Launch Pipeline

One command, 15 seconds, 5 drones airborne-ready

CycloneDDS (default)

t = 0 s   Gazebo world + 5 models spawn
t = 6 s   Per-drone stacks ×5 in parallel
          ├── drone_bridges
          ├── platform_gazebo
          ├── state_estimator
          ├── motion_controller
          └── 5 × behavior servers
t = 10 s  flock_orchestrator  (if enabled)

Zenoh (opt-in)

t = 0 s   zenoh_router daemon
t = 3 s   Gazebo world + 5 models
t = 11 s  tf_static_bridge (QoS workaround)
t = 12 s  Per-drone stacks ×5 in parallel
t = 16 s  flock_orchestrator  (if enabled)

Timer offsets are tuned so action servers are discoverable before orchestrators try to talk to them — the kind of detail that makes scaling to more drones predictable.

ORYNTH / AEROLAB05 — LAUNCH
05 · Operating Modes

Two ways to fly the swarm

MODE A   Teleoperation

Manual control of drone0. Drones 1–4 track it.


# Terminal 1 — full stack
ros2 launch aerolab_simulation \
  swarm_bringup_launch.py

# Terminal 2 — takeoff + formation
ros2 run aerolab_simulation \
  flock_orchestrator

# Terminal 3 — keyboard GUI
ros2 launch aerolab_simulation \
  teleop_launch.py

MODE B   Automated

Full bringup + formation in one command.


# One terminal
ros2 launch aerolab_simulation \
  swarm_bringup_launch.py \
  use_flock_orchestrator:=true

# Optional — attach keyboard teleop
ros2 launch aerolab_simulation \
  teleop_launch.py

Mode B is the one-click demo. Mode A is the debug workflow — each terminal stays attached to one component's log stream.

ORYNTH / AEROLAB05 — MODES
05 · Formation Geometry

Diamond formation

          drone1   (+3m forward)
              ▲
              │
 drone2 ─── drone0 ─── drone3
(+3m L)   (leader)    (-3m L)
              │
              ▼
          drone4   (-3m forward)

How it holds

  • Followers run FollowReferenceBehavior with
    frame_id = "drone0/base_link"
  • Offsets are in drone0's FLU body frame — rotation-aware
  • TF updates propagate every control cycle
  • No polling loop in the orchestrator — fire-and-forget goal

Formation shape is data. Swapping to a V, a line, or a circle is just changing the offset table — no new code path.

Phases 05 – 07

Perception, autonomy, hardware

05 · Perception Pipeline

Human detection that survives measurement, not just demos.

  • Camera payloads on each support drone
  • Perception node per drone → DetectionArray
  • Ground-station fusion node
  • 50–100 m distance-binned validation

06 · Mission Autonomy & Mapping

Search or survey behavior that produces a useful map.

  • BT mission library
  • Search / survey execution and adaptation
  • Mothership LiDAR + multi-drone detections
  • Point cloud registration (ICP / NDT)
  • Semantic 3D map — objects projected into cloud
  • Store-and-forward relay to GCS

07 · Hardware Transition

Same code, real drones.

  • ArduPilot-compatible FCU + companion SBC per drone
  • Separate ArduPilot bringup path via shared mission interface
  • HIL testing before open-air flight
  • Automated pre-flight check gate
  • First outdoor flight: 2-drone formation, 30 s

Moving to hardware keeps the mission layer recognizable, but the ArduPilot SITL and hardware path should be introduced as a separate backend rather than implied by a one-line platform swap.

07 · Risks & Open Questions

What I'd like input on

AreaConcernOptions on the table
Ultralytics Ultralytics runtime licensing must be reviewed before distributing closed-source binaries. (a) Enterprise licence — (b) export to ONNX & run under Apache-licensed runtime
Zenoh maturity Best-on-paper for mesh topologies, QL-2 not QL-1. Smaller ecosystem than DDS. Keep dual-RMW. CycloneDDS default for now, Zenoh opt-in and validated in parallel.
RF range 200–400 m with companion WiFi. Beyond that we need mesh radios. Early hardware uses WiFi. Doodle Labs / Silvus budgeted from Phase 7 onwards.
ArduPilot transition ArduPilot SITL should be introduced as a separate path, not implied by the current AS2 simulation stack. Stay AS2-only in sim. Add ArduPilot SITL and hardware through the shared adapter boundary.
